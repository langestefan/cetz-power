Lines that connect symbols. Wires are not full "symbols" — they have no
label and no style family of their own; their stroke follows the global
`cetz-power.wire.stroke`.

## `wire(a, b)`

Straight line between two coordinates or anchors. A common use is to
drop a tap from a bus to a symbol below it:

<Snippet name="wire-example" />

## Polylines — three or more points

`wire` is variadic — pass any number of positional coordinates and
each consecutive pair is joined by a straight segment. The natural
power-system use is a **transmission line passing through several
substations in series**: one physical conductor, one `wire` call.

<Snippet name="wire-polyline" />

## Relative coordinates

Any positional argument can be `(rel: <vec>)`. CeTZ interprets that
as **"offset from the previous point in the list"** — each step is
relative to where the wire is currently at, not to any named anchor.
This is the natural form for a **feeder that needs to bend around
something** before reaching its load — for example an outgoing
feeder that drops out of the switchgear, runs across the cable
trench, then turns down into a load that's offset from the bus tap:

<Snippet name="wire-relative" />

If you'd rather offset from a *specific anchor* (instead of from
"wherever the wire is right now"), use the longer form
`(rel: <vec>, to: <coord>)`. Useful when a tie cable needs to
**enter and leave each substation perpendicularly** with a known
standoff — pin the approach and landing points to each bus and the
routing follows automatically if the buses move:

<Snippet name="wire-rel-to-anchor" />

In that wire, `(rel: (0.5, 0), to: "b1.mid")` sits 0.5 east of
`b1.mid`, and `(rel: (-0.5, 0), to: "b2.mid")` sits 0.5 west of
`b2.mid` — both anchored to their respective buses regardless of
where the previous step ended up.

## Mixing the forms

Anchor names, absolute coordinates, and relative offsets can all sit
in the same `wire` call. The realistic case for absolute coordinates
in an SLD is a **cable corridor with documented intermediate poles
or joints** — the endpoints stay anchored to each substation's bus
(so they move with the buses), but the route between them is pinned
to specific geographic positions:

## Inline labels — the `label:` argument

Pass `label: <content>` to drop a text caption at the midpoint of
the wire. The midpoint is the lerp-50 % between the **first** and
**last** positional points, so it works the same whether the wire
is a two-point straight line or a multi-point polyline.

<Snippet name="wire-label" />

`label-side:` picks which compass direction off the midpoint the
caption sits on — `"north"` (the default), `"south"`, `"east"`,
`"west"`, or one of the four 45° diagonals. `label-distance:` is
the gap between the wire and the nearest edge of the text:

<Snippet name="wire-label-sides" />

The full set of label-related arguments:

| Argument          | Default | What it controls                                                |
| ----------------- | ------- | --------------------------------------------------------------- |
| `label`           | `none`  | The caption content. `none` = no label drawn.                   |
| `label-side`      | `"north"` | Which side of the wire midpoint the label sits on.            |
| `label-distance`  | `0.15`  | Gap between the wire and the closest edge of the label.         |
| `label-align`     | `"center"` | Horizontal position of the label box relative to the wire midpoint. `"left"` pins the box's left edge at the midpoint (text extends right), `"center"` centres the box, `"right"` pins the right edge (text extends left). Only meaningful when `label-side` is `"north"` or `"south"`. |
| `label-size`      | `7pt`   | Font size for the label.                                        |

For full positioning control, or for labels that aren't tied to a
wire, use the standalone [`note()` helper](#standalone-labels-with-note)
instead.

## Standalone labels with `note()`

`note(pos, body, side: "north")` drops a free-floating caption next
to any coordinate or anchor. It's the building block underneath the
`wire(label: …)` shortcut, exposed for cases where the label doesn't
belong to a wire — labelling a tap point on a bus, annotating an
empty area of the diagram, or pinning a caption to the midpoint
between two anchors via a CeTZ lerp coord:

<Snippet name="wire-note" />

`note` accepts the same compass-side vocabulary as
`wire(label-side: ...)`. The full signature:

```typst
note(
  pos,                         // anchor name, tuple, lerp, …
  body,                        // the caption content
  side: "north",               // compass side of pos
  distance: 0.15,              // gap from pos
  text-align: auto,            // left / center / right; auto keeps cetz's default
  size: 7pt,                   // font size
)
```

## `elbow(a, b, corner: "h")`

L-shaped routing. `corner: "h"` goes horizontally first then vertically;
`corner: "v"` does the opposite.

<Snippet name="wire-elbow" />

## Stroke override

Both accept an optional `stroke:` argument to override the family
default. For `wire`, pass it as a named argument **after** all the
positional coordinates — the polyline form takes any number of
positionals, so the named arg has to come at the end:

```typst
wire(a, b, stroke: 2pt + red)
wire("a.in", (rel: (0, -1)), "b.in", stroke: 2pt + red)
elbow(a, b, corner: "h", stroke: 2pt + red)
```

<Snippet name="wire-stroke" />

To restyle every wire in the canvas at once, set the family default
instead of overriding per call:

```typst
cetz.draw.set-style(cetz-power: (wire: (stroke: 1.4pt + orange)))
```

<Snippet name="wire-family-stroke" />

The `set-style` only affects wires (and elbows); the buses and the
transformer keep their own strokes. See the
[Wires-and-elbows getting-started page](/getting-started/wires/)
for the same pattern in context.

## Colours, dashing, and per-phase coding

`stroke:` accepts CeTZ's full stroke dict —
`(paint: <color>, thickness: <length>, dash: <pattern>)` — so you
can encode information about a connection in its appearance. A few
patterns that come up often in power-system single-lines:

### Three-phase colour coding

Three parallel wires, one per phase, coloured to the IEC convention
(red / yellow / blue). This is how a three-phase distribution feeder
is conventionally drawn when the phases need to be visually
distinguished — e.g. an unbalanced-load study or a connection
diagram for a delta–wye transformer.

<Snippet name="wire-three-phase" />

### Planned / future connections

A grey dashed line is the standard notation for **a connection that
isn't built yet** — a planned tie-line, a contingency switch
position, or any "this isn't here today but might be tomorrow"
element on a network expansion plan. Pass the full stroke dict to
combine colour, thickness, and dash:

<Snippet name="wire-dashed" />

### Voltage-level colour coding

Bus and feeder colours encoded by voltage level — red for HV,
orange for MV, blue for LV — with stroke thickness following the
voltage. A glance at the operations-diagram tells you which part of
the network you're looking at:

<Snippet name="wire-voltage-coded" />

The `stroke:` override works on **buses** the same way as on wires;
this snippet uses it to make each busbar match its feeders'
voltage colour.
